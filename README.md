### Medical Information System (MIS) - Task Tracker API

Микросервис трекера рабочих задач для медицинского персонала. Проект разработан на языке Go с использованием СУБД PostgreSQL по принципам Чистой Архитектуры, SOLID, DTO и CQRS.

### Архитектурные особенности проекта

*   **CQRS:** Логика модификации (Commands) и логика чтения (Queries) полностью разделены по отдельным изолированным компонентам.
*   **SOLID (DIP/ISP):** Тонкие интерфейсы для работы с базой данных объявляются непосредственно на стороне потребителя (в слое бизнес-логики), изолируя инфраструктурные детали.
*   **DTO:** Слой HTTP-доставки защищен от утечек доменных моделей с помощью изолированных DTO-структур с инкапсулированной валидацией входных данных.
*   **Безопасный Enterprise Docker Build:** Применяется многоэтапная сборка (`multi-stage`) на базе защищенного и легковесного образа **Google Distroless**. Контейнер не содержит пакетных менеджеров и утилит командной строки, что минимизирует площадь атаки.

### Требования для локального запуска

Для запуска всей инфраструктуры в одно нажатие на вашем компьютере должны быть установлены:

1.  **Docker Engine** (начиная с версии 20.10+)
2.  **Docker Compose** (современный плагин V2, встроенный в Docker)

*Примечание для пользователей Windows/WSL2:* Убедитесь, что Docker-демон нативно запущен внутри вашей Linux-дистрибуции (Ubuntu) или в Docker Desktop корректно включена интеграция с WSL2 (вкладка Settings -> Resources -> WSL Integration).

### Быстрый запуск (Docker Compose)

Запуск всей цепочки сервисов (База данных -> Автоматические миграции -> API Сервер) осуществляется одной командой из корня проекта:

bash

    sudo docker compose up --build

### Поэтапный процесс развертывания под капотом:

1.  **`med_mis_db`**: Поднимается база данных PostgreSQL. Встроенный `healthcheck` проверяет утилитой `pg_isready` доступность порта.
2.  **`med_mis_migrations`**: Ждет, пока БД перейдет в статус `healthy`. Затем утилита `golang-migrate` автоматически накатывает SQL-миграции схемы таблиц и индексов и завершает свою работу с кодом `0`.
3.  **`med_mis_api`**: Ждет успешного завершения миграций, компилирует статический бинарник Go 1.26.1, запускает его в защищенной среде Distroless и открывает порт `8080` для внешних запросов.

Для остановки приложения с полной очисткой кэша контейнеров и томов данных выполните:

bash

    sudo docker compose down -v

После успешного запуска API доступно по адресу `http://localhost:8080`. Вы можете протестировать эндпоинты через любой HTTP-клиент (Postman, Insomnia) или через терминал с помощью `curl`.

*Запрос возвращает задачи постранично в объекте `data` вместе с метаданными `pagination`.*
```bash
curl "http://localhost:8080/api/v1/tasks?status=NEW&page=1&limit=20"
```

#### 3. Получение задачи по ID (GET)
```bash
curl http://localhost:8080/api/v1/tasks/1
```

## Шаг 2. Интеграция тегов задач и управление справочником

В рамках второго этапа в систему добавлен функционал слабосвязанных метаданных (тегов) с разделением жизненных циклов ресурсов (Lifecycle Split) согласно RESTful-принципам. Существующий репозиторий задач `TaskRepository` полностью закрыт от изменений (соблюден принцип Open-Closed / OCP).

### Новые архитектурные решения Шага 2

*   **Богатая доменная модель (Value Object):** Тег представлен доменной сущностью `Tag` со встроенными инвариантами валидации. Бизнес-логика (`tag.CanDelete()`) изолирована внутри домена и защищает обязательные системные теги клиники (`отчетность`, `операции`, `звонок`) от удаления.
*   **Атомарные интерфейсы Use Case (ISP):** Контракты тегов декомпозированы на тонкие независимые интерфейсы (`TaskTagsSyncer`, `TaskTagsBulkViewer`, `TagRegistryLister`, `TagRegistryRemover`).
*   **Исключение N+1 при чтении (Batch Fetching):** Склеивание задач со списками тегов Many-to-Many происходит на слое Use Case за два быстрых пакетных запроса к СУБД с использованием оператора `ANY($1)`. Это гарантирует 100% сохранение работоспособности SQL-пагинации (`LIMIT/OFFSET`).
*   **Отказоустойчивость записи (Silent Fail):** Привязка тегов к задачам в командах создания и обновления изолирована в транзакции репозитория тегов и выполняется через атомарный `INSERT ... SELECT`. Если тег отсутствует в справочнике МИС, операция тихо и без ошибок пропускается.
*   **Каскадное администрирование:** Удаление кастомного тега из справочника администратором автоматически вычищает его изо всех задач клиники средствами СУБД PostgreSQL (`ON DELETE CASCADE`).

## Шаг 3. Настройки периодичности и виртуальный календарь задач

В рамках третьего этапа в систему добавлен функционал планирования периодических задач медицинского персонала клиники (ежедневные, ежемесячные, на конкретные даты, четные/нечетные дни) с поддержкой изолированного трекинга состояний каждого отдельного дня выпадения задачи.

### Новые архитектурные решения Шага 3

*   **Производительный виртуальный расчет (Light Virtual):** Для предотвращения «проблемы бесконечности» система полностью отказывается от физической генерации миллионов строк-задач в СУБД на годы вперед. В таблице `tasks` сохраняется одна базовая запись-шаблон с nullable-параметрами расписания. Размножение задач на лету происходит в оперативной памяти слоя Use Case в рамках запрашиваемого фронтендом временного окна `[due_date_from, due_date_to]`.
*   **Изоляция состояний экземпляров (Task Executions Log):** Для решения проблемы независимого статуса конкретного дня повторения задачи введена компактная таблица-лог `task_executions`. Изменение статуса дня (например, отметка «Выполнено» врачом) выполняется через изолированный атомарный `UPSERT` (`INSERT ... ON CONFLICT UPDATE`) и привязывается к дате календаря, не затрагивая основной шаблон.
*   **Инкапсуляция календарной математики в Домене (Value Object):** Вся бизнес-логика валидации типов повторений, расчета интервалов и проверки четности чисел месяца вынесена в чистый доменный метод `IsMatch` внутри Value Object `TaskRecurrence`. Логика полностью отвязана от инфраструктуры и СУБД.
*   **Гибридная пагинация в памяти (In-Memory Pagination):** В связи с тем, что окончательный размер списка календаря вычисляется динамически на лету, механизмы `LIMIT` и `OFFSET` перенесены из SQL-запроса на уровень Use Case (слайсинг сформированного виртуального слайса). Пакетный сбор тегов (`FetchTagsForTasks`) оптимизирован и выполняется до этапа размножения, снижая нагрузку на СУБД и полностью устраняя проблему N+1.
*   **Изолированное операционное действие:** Внедрен специализированный эндпоинт `POST /api/v1/tasks/{id}/executions` для фиксации факта выполнения конкретного дня. Это сохраняет чистоту REST-метода `PUT`, предназначенного исключительно для полного административного изменения полей шаблона задачи.

### Тестирование функционала Шага 3 (Smoke Tests)

#### В. Работа с периодическими задачами и календарем

**1. Создание периодической задачи-шаблона (POST)**
*Передается блок `recurrence` с типом повторения (в данном примере: ежедневно, каждый 1 день).*
```bash
curl -X POST http://localhost:8080/api/v1/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Ежедневный обход палат",
    "description": "Проверить показатели пациентов в реанимации",
    "due_date": "2030-06-22T08:00:00Z",
    "tags": ["операции"],
    "recurrence": {
      "type": "DAILY",
      "interval": 1
    }
  }'
```

**2. Получение виртуального календаря за указанный период (GET)**
*Юзкейс на лету размножит шаблон по дням внутри диапазона `due_date_from` и `due_date_to`. Каждая виртуальная задача вернет признак `is_recurring: true`.*
```bash
curl "http://localhost:8080/api/v1/tasks?due_date_from=2030-06-22T00:00:00Z&due_date_to=2030-06-25T23:59:59Z&page=1&limit=20"
```

**3. Изолированная отметка выполнения конкретного дня повторения (POST)**
*Фиксирует статус `DONE` строго для указанной даты в таблице лога исполнений, не ломая статус шаблона на другие дни.*
```bash
curl -X POST http://localhost:8080/api/v1/tasks/1/executions \
  -H "Content-Type: application/json" \
  -d '{
    "date": "2030-06-22T00:00:00Z",
    "status": "DONE"
  }'
```

**4. Полное обновление параметров расписания задачи-шаблона (PUT)**
*Перезаписывает метаданные периодичности в СУБД (например, переводит на ежемесячный повтор 15-го числа).*
```bash
curl -X PUT http://localhost:8080/api/v1/tasks/1 \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Ежедневный обход палат (Обновленное расписание)",
    "description": "Проверить показатели пациентов в реанимации",
    "due_date": "2030-06-22T08:00:00Z",
    "status": "IN_PROGRESS",
    "tags": ["операции"],
    "recurrence": {
      "type": "MONTHLY",
      "day_of_month": 15
    }
  }'
```
